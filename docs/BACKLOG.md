## P2（未来，当前 milestone 不挤占）

- **inverse-tonemap-hable-impl** — Cycle 24 landed `Linear` in `src/compose/inverse_tonemap_kernel.cpp:apply_inverse_tonemap_inplace`; `Hable` algo still returns ME_E_UNSUPPORTED at line 70 (gated by an explicit early-return). Real Hable inverse needs linear-light float buffers: the curve `f(x) = ((x*(A*x+C*B)+D*E)/(x*(A*x+B)+D*F))-E/F` is invertible only on the [0, white_scale] domain, and byte-domain inverse rounding loses enough precision that the round-trip fails VISION §3.1 byte-identity. **方向：** when a 16-bit (or RgbaF32) working buffer arrives in M11+, replace the Hable UNSUPPORTED branch with the proper linear-light inverse: read SDR byte → linearize via inverse-sRGB-EOTF → numerical inverse of Hable curve via Newton–Raphson on the [0, white_scale] domain → encode to PQ / HLG / linear-light HDR buffer. Until then leave UNSUPPORTED as the deterministic answer for Hable. Milestone §M11-followup，Rubric §5.2。
- **ml-ship-path-model-blazeface** — `grep -rni 'blazeface\|face_landmarks_v2\|selfie_seg' src/ include/` returns empty (only test fixture strings hit). M11 exit criterion 6 (docs/MILESTONES.md:139) requires "≥2 ship-path models running through the runtime + reference parity within ε". BlazeFace + portrait_segmentation are the Apache-licensed candidates per the criterion text. **方向：** when CoreML / ONNX runtimes are wired (above two bullets), add `examples/14_ml_face_landmark/` + `tests/test_inference_blazeface.cpp` that loads BlazeFace through the model-fetcher API + runs inference + asserts CoreML-vs-ONNX-CPU parity within ε. Model bytes NOT bundled (license-clean: hosts supply via fetcher); the test uses a synthetic deterministic test fixture or a host-environment-supplied blob. Milestone §M11，Rubric §5.1。
- **encode-hevc-main10-kvazaar-source-build** — `src/io/kvazaar_hevc_encoder.cpp:88` calls `kvz_api_get(8)` exclusively; the homebrew Kvazaar bottle's `kvazaar.h` defaults to `KVZ_BIT_DEPTH 8` and the prebuilt library doesn't expose a 10-bit `kvz_pixel` ABI. `cfg->input_bitdepth = 10` is accepted by `encoder_open` but the `picture_alloc` buffer geometry stays 8-bit. M10 exit criterion 3 (docs/MILESTONES.md:120) requires Main 10 from the SW path for HDR PQ/HLG output; this cycle ships Main 8-bit only so HDR Main 10 stays on the VideoToolbox HW path. **方向：** replace `pkg_check_modules(kvazaar)` with `FetchContent_Declare(kvazaar URL ...)` + `cmake -DKVZ_BIT_DEPTH=10` so the engine links its own 10-bit-built libkvazaar.a; rewrite the encoder TU's plane-copy loops to pass uint16_t buffers (interpreting `kvz_pixel*` as `uint8_t*` view of the same bytes). Add a fixture YUV420P 10-bit gray frame and a round-trip test that decodes the produced HEVC Main 10 and verifies pixel domain matches within ε. Milestone §M10-followup，Rubric §5.2。
- **encode-hevc-output-sink-runtime-sw-dispatch** — `src/orchestrator/output_sink.cpp` `is_video_aac_spec` now routes `(hevc, aac)` to `VideoAacSink`, but the encoder picked inside `me::orchestrator::detail::open_video_encoder` (`src/orchestrator/reencode_video.cpp:34-37`) is `hevc_videotoolbox` unconditionally on Apple — `KvazaarHevcEncoder` is linked when `ME_HAS_KVAZAAR` but never selected by the pipeline. M10 exit criterion 3 wants the SW fallback to kick in when VideoToolbox HEVC is unavailable (Linux / Windows hosts) or when explicitly requested. **方向：** in `open_video_encoder`'s `is_hevc` branch, probe `avcodec_find_encoder_by_name("hevc_videotoolbox")` first; on null and `ME_HAS_KVAZAAR`, build a parallel encode-frame loop that pumps YUV420P frames through `KvazaarHevcEncoder` and emits packets via `av_packet_from_data` (the existing libavcodec encode loop is in `encode_video_frame`; the SW path needs its own — likely a sibling helper `encode_video_frame_kvazaar` that bypasses `avcodec_send_frame`). Add a host-side opt-in via `ME_OUTPUT_SPEC_VIDEO_CODEC=hevc-sw` or similar so callers can force the SW path even when HW is available (useful for cross-host determinism comparisons). Milestone §M10-followup，Rubric §5.2。
