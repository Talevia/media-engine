# Backlog

`iterate-gap` skill 的唯一任务源。档内按出现顺序挑，不重排。

**格式**：`- **<slug>** — <Gap：现状 / 痛点>。**方向：** <期望动的东西>。Milestone §M<x>，Rubric §5.<y>。`

`<slug>` 是 kebab-case、一眼看出改什么。`debt-*` 前缀表示技术债。

**本文件只由 iterate-gap skill 写**：新任务通过 repopulate 产出；每轮处理完的 bullet 在当轮的 `feat(...)` commit 里删掉（决策理由写进 commit body，详见 `.claude/skills/iterate-gap/SKILL.md` §7）。手工修改请走 `docs(backlog):` 独立 commit 并说明理由。

**Repopulate 纪律（2026-04-23 后）**：每条 bullet 的 `Gap` 部分必须引用 `grep` 查到的具体 `path:line` 证据，不凭印象写"静默丢掉"/"没覆盖"等转述（连续 3 cycle 踩这坑；现在 SKILL.md §R 硬性要求）。

## P0（必做，阻塞当前 milestone）

- **effect-gpu-color-correct** — 首个真实 GPU effect。`grep 'ColorCorrect\|Brightness\|Contrast\|Saturation' src` 空。M3 exit criterion "≥ 3 个 GPU effect (blur / color-correct / LUT)" 中最简单的一个（单 fragment shader、3 个标量 uniform，无 kernel 邻域采样）。**方向：** 新 `src/effect/color_correct_effect.{hpp,cpp}` : `GpuEffect`，embedded GLSL-ES fragment shader `fs_color_correct.sc`（brightness / contrast / saturation 各一 uniform float）；先不走 bgfx shaderc 预编译（会把 BGFX_BUILD_TOOLS 打开，重新 ~3min 编译）——用 `bgfx::ShaderHandle bgfx::createEmbeddedShader` + 手写 SPIR-V 字节数组，或者 runtime `bgfx::createShader` 用 `BGFX_RESOURCE_PATH` 下的编译产物（一次性 commit 编译好的 .bin）。实装 tests/test_color_correct_effect.cpp：创建 1×1 红色 src framebuffer，brightness=0.5，readback 应为灰色（0.5, 0, 0）。Milestone §M3，Rubric §5.3。

## P1（强烈建议，M3 主线 / 跨 milestone debt）

- **effect-gpu-blur** — 依赖 `effect-gpu-kernel-infra` 落地后的第 2 个 GPU effect。`grep 'BlurEffect\|class.*Blur\b' src` 空。**方向：** 新 `src/effect/blur_effect.{hpp,cpp}` : GpuEffect，two-pass separable Gaussian（水平 + 垂直），kernel radius 参数化（uniform int），shader `fs_blur_h.sc` + `fs_blur_v.sc`。two-pass 需要中间 framebuffer——借 GpuEffectChain 的 ping-pong scratch。Milestone §M3，Rubric §5.3。
- **effect-gpu-lut** — 第 3 个 GPU effect，闭合 M3 exit criterion "≥ 3 个 GPU effect"。`grep 'LutEffect\|class.*Lut\b' src` 空。3D LUT 采样是色彩分级 baseline。**方向：** 新 `src/effect/lut_effect.{hpp,cpp}` : GpuEffect；接受一个 33×33×33 RGB LUT（1D texture 编码或真 3D texture，后者 Metal 支持），shader 做 trilinear 采样。LUT 源从文件路径加载（.cube 文件 parser，或直接从 OCIO 黑箱取）。先实装 .cube 路径；OCIO integration 单独走。Milestone §M3，Rubric §5.3。
- **effect-chain-gpu-pass-merge** — `src/effect/effect_chain.hpp:55-57` 的 `EffectChain::apply` 是裸 for 循环调用每个 Effect::apply，无合并。M3 exit criterion "EffectChain 能把连续 ≥ 2 个像素级 effect 合并成单 pass" 明文要求 GPU 路径合并。**方向：** 新 `me::effect::GpuEffectChain::compile()` — 检测同形 effect（color-correct + color-correct、或更一般地：所有 effect 标 `is_per_pixel() == true` 且读写样式兼容）并生成 fused fragment shader：把两个 effect 的 fragment main 串联、uniform buffer 合并、layout 重打包。Fallback: non-fuseable effects 走独立 pass。test: 2 个 color-correct chain 跑 profiler 确认只有 1 次 draw call。Milestone §M3，Rubric §5.1。
- **compose-sink-gpu-branch** — `grep -n gpu_backend src/orchestrator` 空；`src/orchestrator/compose_sink.cpp` 是硬编码 CPU 路径（`me::compose::affine_blit`, `me::compose::alpha_over` 都是 CPU 函数）。`me_engine.gpu_backend` 已在 (5566bea) 入驻 engine_impl，但 ComposeSink 没拿它。**方向：** `make_compose_sink(...)` 接受 `const me::gpu::GpuBackend*` 参数（engine 调 `gpu_backend.get()`）。`ComposeSink::process` 在循环开始检测 `gpu && gpu->available() && !strncmp(gpu->name(), "bgfx-", 5) && strcmp(gpu->name(), "bgfx-Noop") != 0` → 走 GPU compose 路径（未来落地）；否则保留 CPU。这次 cycle 只加参数通路 + 判定函数，不真走 GPU compose。Milestone §M3，Rubric §5.3。
- **timeline-loader-effect-parse** — `src/timeline/timeline_loader.cpp:184-185` 显式 reject `clip.effects`：`require(!clip.contains("effects") || clip["effects"].empty(), ME_E_UNSUPPORTED, "phase-1: clip.effects not supported")`。Timeline IR `src/timeline/timeline_impl.hpp` 也无 effect 字段。M3 要求 timeline 里能声明 GPU effects。**方向：** (1) 扩 `me::Clip` struct 加 `std::vector<EffectSpec> effects;` 其中 `EffectSpec` 是 typed tagged union (`kind: ColorCorrect` → `ColorCorrectParams`, `kind: Blur` → `BlurParams`, ...)。(2) 扩 `loader_helpers.cpp` 加 `parse_effect_spec(const json&)` schema 校验。(3) 删 timeline_loader.cpp:184 的 reject 路径，改成 loop 解析每条 effect。(4) schemaVersion 保持 1（旧 JSON 照常解，effects 字段 default 空）。Milestone §M3，Rubric §5.2。
- **gpu-render-1080p60-bench** — `grep -rn 'benchmark\|1080p\|60fps' tests examples` 空。M3 exit criterion "1080p@60 可实时渲染带 3-5 个 GPU effect 的 timeline" 需要 bench harness + 性能闸。**方向：** 新 `bench/bench_gpu_compose.cpp`（build target `bench_gpu_compose` 只在 `ME_BUILD_BENCH=ON` 且 `ME_WITH_GPU=ON` 时开）。跑 N 帧 1080p × (color-correct + blur + lut) chain，`std::chrono::high_resolution_clock` 测端到端时间，断言 avg frame time < 16.67 ms（60 fps budget）。Build-time smoke：跑 10 帧即可（CI 时间预算）；release 跑 600 帧更稳。Milestone §M3，Rubric §5.3。
- **debt-split-compose-sink-process** — `tools/scan-debt.sh` 命中长文件：`src/orchestrator/compose_sink.cpp` 574 行 > 400 阈值。真正密度源：`ComposeSink::process()` 从 81 行到 492 行，一个 400+ 行的方法包含 demux-range 校验 / encoder-mux setup / AudioMixer setup / 逐 clip decoder 状态 / 主 decode-compose 循环 / drain 收尾。**方向：** 按"每个 process 阶段" 抽出 free function（类似 `compose_audio.{hpp,cpp}` / `compose_transition_step.{hpp,cpp}` 的先例）：`setup_compose_decoders()` / `run_compose_decode_loop()` / `drain_compose()`。目标 < 300 行。Milestone 跨（M3 debt），Rubric §5.1。
- **bgfx-render-thread-pin** — `src/gpu/bgfx_gpu_backend.hpp:21` 注释明文："bgfx itself is single-threaded on the API thread; compose kernels issuing bgfx calls must pin to a dedicated render thread. That's future work — not yet wired." 今天 BgfxGpuBackend 仅被 engine-create 一个线程访问（ctor/dtor），没问题；但一旦 compose kernel 从 scheduler worker 线程调 bgfx，就会踩 bgfx 的单线程 contract。**方向：** 新 `src/gpu/render_thread.hpp` — 一个 MPSC queue 收 `std::function<void()>` lambdas，后台单线程拉出来执行（bgfx::frame 也在这里）。`BgfxGpuBackend` 构造时启动，析构时 join。future Gpu effect submit 调用走 `bgfx_backend->submit_on_render_thread(...)` 进 queue。Milestone §M3，Rubric §5.3。

## P2（未来，当前 milestone 不挤占）

- **codec-pool-real-pooling** — `src/resource/codec_pool.hpp:6` 注释: "encoder reuse (the "pool" in the name) is deferred"；`codec_pool.cpp` 只 `++live_count_` / `--live_count_`。`reencode-multi-clip` N-segment 开独立 AVCodecContext（`reencode_segment.cpp:112`），但每段 open_decoder 只 ~ms，没 profile 证据表明是瓶颈。**方向：** 等 M4 多段音频或 benchmark 证实瓶颈再加 `get_or_make_decoder(codec_id, codecpar)` + `avcodec_flush_buffers` pool 路径。Milestone §M4-prep，Rubric §5.3。
- **async-job-base** — 当前只有 `me_render_start` 一个异步入口，worker→caller 的 error propagation 走 `Job::err_msg` + `me_render_wait` 中转。等第二个异步 API 落地（大概率 M6 frame-server async preview）时抽 `AsyncJobBase` 收口。Milestone §M6-prep，Rubric §5.2。
- **me-output-spec-typed-codec-enum** — `docs/PAIN_POINTS.md` 2026-04-22 记录：`me_output_spec_t.video_codec` / `audio_codec` 是 `const char*`，每加一个 codec 就要多一个 `is_xxx_spec` helper + 一段 `strcmp` 分支；`video_bitrate_bps` 跨 codec 共享不分。现有两 codec（passthrough, h264）不痛；M3–M4 落第 3、4 个 codec 时是评估"C ABI 引入 typed option union"的决策点。**方向：** 跨 C ABI 的 typed option union 设计（`me_video_codec_t` enum + per-codec `me_h264_opts_t`、`me_aac_opts_t` struct + `me_output_spec_t` 带 tagged pointer）。重大 ABI 演进，不在 M2 scope。Milestone §M3-prep，Rubric §5.2。
- **debt-gpu-backend-noop-fallback-test** — `src/gpu/bgfx_gpu_backend.cpp:43-47` 的两阶段重试（Count → Noop）只在 auto-pick 失败时跑到 Noop；dev macOS 上 Metal 头less init 现在稳定成功（0×0 分辨率修复后），Noop 分支没 CI 覆盖。`grep 'Noop\|name.*bgfx-Noop' tests` 空。真到某个 driver 拒绝 headless init 才暴露这条分支的 bug。**方向：** 测试新增一条 strongly-force-Noop 路径——如提供 `ME_GPU_FORCE_NOOP=1` 环境变量让 BgfxGpuBackend 跳过 auto-pick 直接进 Noop；对应 test 断言 `name() == "bgfx-Noop"` 且 `available() == true`。也验证 dtor 在 Noop 初始化后能干净 shutdown。Milestone §M3-debt，Rubric §5.3。
