- **debt-bindings-kn-toolchain-port** — Cycle 83 stood up `kn_cinterop_smoke` ctest gated on gradle <9 because `kotlin("multiplatform") version "2.0.20"` (pinned in `bindings/kotlin-native/example/build.gradle.kts:23`) breaks against Gradle 9 (DefaultArtifactPublicationSet removed). Milestone §M7-debt (cross)，Rubric §5.5。
- **debt-cache-invalidate-asset-graph-hash** — `src/api/render.cpp:46-47` already documents this: `me_cache_invalidate_asset` was originally written when DiskCache keys had shape `<asset_hash>:<src_t>` so prefix-matching the asset hash dropped derived frames; the move to graph-content-hash keys (`g:<hash>`) made that prefix match a no-op. Confirmed empirically by `examples/12_cache_invalidate` this cycle: after `invalidate_asset`, `entry_count` (AssetHashCache size) drops as expected but DiskCache entries derived from the same asset still serve the next render — `miss_count` does not climb. Hosts that need real eviction must call `me_cache_clear` (universal escape hatch). **方向：** either (a) embed each contributing asset's content_hash into the graph-content-hash payload + walk DiskCache keys for asset-hash substring match, or (b) maintain a side index `asset_hash → set<graph_hash>` populated on every DiskCache put and consulted by invalidate_asset. (a) is simpler but iterates the full DiskCache; (b) is per-asset O(1) but adds a write-amplification step. Pick after measuring DiskCache typical size in M11+ workloads. Milestone §M6-debt (cross)，Rubric §5.4。
- **inverse-tonemap-effect-impl** — `src/compose/inverse_tonemap_kernel.cpp:apply_inverse_tonemap_inplace` returns `ME_E_UNSUPPORTED` today (registered-but-deferred per `inverse-tonemap-effect-stub`, M10 exit criterion 6 second half). Real SDR→HDR expansion is non-deterministic by construction (heuristics for invented highlight detail; neural networks for shadow-texture reconstruction); landing it requires either (a) labelling the path explicitly non-deterministic per VISION §3.1, behind a `ME_ALLOW_NONDETERMINISTIC_EFFECTS` opt-in; or (b) a deterministic linear-expansion baseline (Linear algo) that works without invention but only for HDR-output scenarios where the SDR input was downsampled from HDR earlier in the pipeline. Two algos already declared: `Linear` (deterministic identity-style expansion → `target_peak_nits / 100` linear scale) and `Hable` (the inverse of `TonemapEffectParams::Algo::Hable` — invertible only on the [0, white_scale] domain). Start with Linear since it's deterministic. **方向：** drop the unconditional UNSUPPORTED return, implement Linear as `out = clamp(in/255 * (target_peak_nits/100), 0, 255)` (rounded back to byte; loses HDR range as expected — placeholder until a 16-bit working buffer exists in M11+); leave Hable as a separate cycle that lands once linear-light float buffers arrive. Milestone §M10-followup，Rubric §5.2。
- **debt-player-rate-audio-tempo** — `src/orchestrator/player.cpp:Player::play` (cycle 20) admits forward variable rate ∈ [0.5, 2.0] for video-only WALL-master timelines but rejects `rate ≠ 1.0` whenever `has_audio_track_` is true OR master clock is `ME_CLOCK_AUDIO` (UNSUPPORTED). Rationale: audio output runs at the timeline's native sample_rate; without time-stretching, audio would desync against scaled video. `src/audio/tempo.hpp:44` already wraps SoundTouch but `audio_producer_loop` (player_audio_producer.cpp) never instantiates a TempoStretcher. **方向：** plumb rate (read from PlaybackClock or stored on Player) into audio_producer_loop's per-chunk emit path; instantiate TempoStretcher with the configured rate and feed it AudioMixer output before push to the host's audio_cb. Tests: extend test_player's audio-rate case to ME_OK + a sample-count drift bound that proves audio matches scaled video pacing. Milestone §M8-followup / cross M4 (audio chain)，Rubric §5.5。
